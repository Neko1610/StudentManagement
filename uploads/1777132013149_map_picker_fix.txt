Gap Analysis

This codebase is a decent prototype, not a production-ready e-commerce system yet. The biggest problems are insecure backend authorization, checkout integrity gaps, several broken API contracts between clients and backend, and a lot of “looks implemented in UI but not really implemented in business logic.”

🔥 Critical

Product and Category admin writes are publicly exposed. In SecurityConfig.java (line 30), /api/products/** and /api/categories/** are permitAll(), which means anyone can create/delete catalog data because ProductController.java (line 97) and CategoryController.java (line 36) do not enforce admin checks.
Fix: split public read routes from admin write routes, e.g. GET /api/products/** public, POST/PUT/DELETE /api/admin/products/** admin only.

Voucher CRUD and variant mutation are not admin-protected. VoucherController.java (line 32) and VariantController.java (line 25) rely only on global auth, so any logged-in customer can modify vouchers and variants.
Fix: move them under /api/admin/** or call userService.requireAdmin() in every write endpoint.

Checkout bypasses voucher validation entirely. In OrderService.java (line 83), checkout only checks voucher existence, not active, expiryDate, or minOrder, even though VoucherService.java (line 54) already has that logic.
Fix: reuse voucherService.applyVoucher(...) or a shared validator before calculating discount.

Checkout never validates stock and never decrements stock. OrderService.java (line 62) calculates totals only; it does not reject oversold variants or reduce inventory after order creation.
Fix: inside the transaction, check variant.getStock() >= qty, then variant.setStock(variant.getStock() - qty) and save.

Cart operations are vulnerable to IDOR and broken ownership logic. CartService.java (line 63) deletes by cart id only, and CartService.java (line 67) updates by cart id only, without verifying the item belongs to the authenticated user.
Fix: query by id + userId, not plain id.

Shipping fee calculation leaks other users’ addresses. ShippingController.java (line 29) loads any addressId directly, without checking the address email against the JWT user.
Fix: resolve address with findByIdAndEmail(addressId, currentEmail).

Secrets are committed in source. application.properties (line 2) contains DB credentials, JwtUtil.java (line 13) hardcodes the signing key, and firebase-service-account.json exists in src/main/resources.
Fix: move all secrets to environment variables or external secret storage, rotate them, and remove committed credentials from git history.

Mobile cart update/delete is calling the wrong id. cart_screen.dart (line 76) sends item.variantId to update quantity, but backend expects cart item id. cart_item_widget.dart (line 21) also uses item.variantId for update/delete.
Fix: always call cart APIs with item.id, not variantId.

Mobile voucher apply endpoint is wrong, so voucher input is effectively broken. cart_service.dart (line 61) calls /api/voucher/apply, but backend route is /api/vouchers/apply in VoucherController.java (line 50).
Fix: change the app path to /api/vouchers/apply.

Admin product edit is broken by contract mismatch. Dashboard calls PUT /products/{id} in productService.ts (line 46), but backend has no @PutMapping("/{id}") in ProductController.java (line 39).
Fix: implement update endpoint or change dashboard behavior.

Admin user list is broken by route mismatch. Dashboard calls /users in userService.ts (line 22), while backend exposes /api/admin/users in AdminUserController.java (line 13).
Fix: point dashboard to /admin/users.

⚡ Important

Dashboard auth is not really admin auth. authService.ts (line 63) accepts any pasted JWT, and userService.ts (line 17) force-labels the profile as 'Administrator'. A normal user token can enter the dashboard shell.
Fix: verify role from backend and block non-admin users before rendering admin routes.

Product image handling is inconsistent. Backend accepts product.image on create, but response mapping in ProductController.java (line 169) ignores it and derives the product image from the first variant image. Products without variants can come back blank.
Fix: fall back to p.getImage() when no variant image exists.

Mobile route config is incomplete. main.dart (line 55) only registers /login, /register, /home, but product_detail_screen.dart (line 194) navigates to /cart.
Fix: either add a /cart route or navigate with MaterialPageRoute.

Category screen can spin forever on empty product results. category_screen.dart (line 70) shows a loading spinner whenever products.isEmpty, instead of checking provider.isLoading.
Fix: branch on isLoading, then render an empty state when loading is false.

Order detail screen can hang forever on API failure. order_detail_screen.dart (line 39) catches errors but never sets isLoading = false.
Fix: set error state and stop loading in catch.

Order detail widgets don’t match backend payload keys. order_address.dart (line 26) expects receiverName, which backend never sends. order_summary.dart (line 14) expects shipping and tax, but backend sends shippingFee and no tax field.
Fix: align widget fields to backend DTOs.

Checkout uses hardcoded phone and inconsistent currency math. checkout_screens.dart (line 95) sends "0123456789" instead of real user data, and checkout_screens.dart (line 72) divides shipping by 24000 while other values are treated like dollars in some places and VND in others.
Fix: define a single currency strategy and use actual phone from profile/address.

Analytics are fake. DashboardPage.tsx (line 14) uses hardcoded chartData, and DashboardPage.tsx (line 82) says “All systems ready” regardless of backend state.
Fix: replace with real aggregated APIs for revenue, order counts, conversion, top products.

Search/filter/sort are far from production level. Backend product search is in-memory in ProductController.java (line 68), with no sort, price filters, stock filter, brand filter, or pagination.
Fix: move filtering/sorting to repository queries with pageable endpoints.

No media upload implementation. There is no Cloudinary/S3/local upload API; dashboard just stores image URLs or base64 strings.
Fix: add a proper media service and store media ids/URLs, not raw base64 in business entities.

No reviews and ratings anywhere. There are no review models, endpoints, admin moderation tools, or mobile review UIs.
Fix: add Review, rating, verifiedPurchase, moderation, aggregate score fields.

🧩 Optional / Nice To Have

Wishlist is app-local only via provider state, not persisted server-side.
Payment methods screen in mobile is just UI scaffolding; no saved cards or wallet integration.
Profile stats contain hardcoded voucher/points values in profile_stats.dart (line 51).

No customer notifications, email confirmations, refund flow, or return flow.

Bad Practices

Direct entity exposure from controllers instead of response DTOs in several places.
RuntimeException everywhere instead of typed exceptions and proper HTTP responses.
Debug logging of tokens and headers in JwtFilter.java (line 50).
spring.jpa.hibernate.ddl-auto=update in production config in application.properties (line 6).
No @Valid, no request validation, no centralized exception handler.
N+1 query patterns in ProductController.java (line 167) and OrderService.java (line 212).
